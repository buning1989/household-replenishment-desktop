- 做饭频率较高，洗衣频率中等

| residentCount | 3 |
| children | schoolAge |
| pets | cat |

不包含任何姓名、地址、手机号等个人隐私信息。

## 4. Smoke 依赖

### 4.1 "猫砂还能用多久"

预期回答依据：

- 猫砂存在且只有一个（id: demo-cat-litter）
- lastRestockedAt: 2026-07-03
- cycleDays: 17
- depletionAt = 2026-07-03 + 17 = 2026-07-20
- 参考日期 2026-07-12 距离 2026-07-20 还有 8 天
- Agent 应回答"还剩约 8 天"

### 4.2 "最近一次买猫砂是什么时候"

预期回答依据：

- 猫砂 history 最新记录：at = 2026-07-03
- Agent 应回答"2026 年 7 月 3 日"

### 4.3 洗衣液为什么会被判定为已存在

- 洗衣液在 Demo items 中存在（id: demo-laundry-detergent, name: "洗衣液"）
- 当用户输入"帮我加个消耗品叫洗衣液"时，Agent 的 buildLocalDraftFromText 会匹配到已存在的"洗衣液"
- Agent 应进入 navigate 状态（而非 createItem proposal）

### 4.4 "今天买了 1 瓶洗衣液，28 元"为什么匹配为补货

- 洗衣液已存在
- Agent 的 buildLocalDraftFromText 解析出 itemName="洗衣液"、qty=1、unit="瓶"、price=28
- 匹配到已存在的洗衣液 item → 生成 restock draft（而非 createItem）

### 4.5 洗衣凝珠为什么可以进入新建流程

- 洗衣凝珠在 Demo items 中不存在
- 洗衣凝珠 ≠ 洗衣液（去重边界测试，参考 tests/agent-403-createitem-dedup.test.mjs G.3）
- 当用户输入"帮我加个消耗品叫洗衣凝珠"时，Agent 应生成 createItem proposal
- 当用户输入"就这么记"时，Agent 应确认创建

### 4.6 平台可选字段覆盖

Demo 数据覆盖以下平台组合：

- 有金额、有平台：猫砂(h1: 65元/淘宝)、洗衣液(32元/京东)、猫粮(120元/京东) 等
- 有金额、无平台：猫砂(h3: 68元/无)、抽纸(h3: 35元/无)、垃圾袋(h2: 16元/无)
- 有评价信息：宠物擦脚湿巾(质量不错)、猫粮(猫咪爱吃)

## 5. 执行说明

### 5.1 如何写入 Demo 数据

```bash
npm run demo:seed
```

该命令将 Demo 数据写入 Electron 的 userData 目录下的 `reminder-state.json`。

安全规则：

- 仅用于空数据库（无 items 的 state 文件）
- 如果已有业务数据，停止并提示，不覆盖
- 不承担重置和覆盖功能

### 5.2 如何验证数据

```bash
npm run demo:verify
```

该命令读取已写入的 state 文件，检查所有核心约束是否成立，包括：

- 商品总数 = 15
- 补货记录总数在 30-45 范围内
- 猫砂存在且只有一个，最近补货日期为 2026-07-03，数量 2 袋，金额 68 元
- 洗衣液存在且只有一个
- 洗衣凝珠不存在
- 宠物擦脚湿巾、抽纸、垃圾袋存在
- 不存在重复商品
- 不存在未来补货记录
- 所有补货记录均能关联到商品
- 猫砂状态为 warning（即将需要关注）
- 猫砂预计剩余 5-8 天

### 5.3 数据写入位置

| 平台 | 路径 |
|---|---|
| macOS（开发） | `~/Library/Application Support/household-replenishment-desktop/reminder-state.json` |
| macOS（打包） | `~/Library/Application Support/403家庭管家/reminder-state.json` |
| Windows（开发） | `%APPDATA%/household-replenishment-desktop/reminder-state.json` |
| Windows（打包） | `%APPDATA%/403家庭管家/reminder-state.json` |

### 5.4 如何确认没有覆盖真实用户数据

1. `demo:seed` 启动时先读取现有 state 文件
2. 如果 items 数组非空，立即停止并提示
3. 只在 items 为空（或文件不存在）时才写入 Demo 数据
4. 原有数据不会被修改或删除

如需重置并恢复 Demo 数据（覆盖现有数据），请使用"一键恢复 Demo state"功能（下个任务实现）。

### 5.5 注意事项

- Demo 数据的参考日期为 2026-07-12。随着时间推移，商品的剩余天数会变化，但核心数据（历史记录、周期）不变。
- 如果应用的 localStorage 中有较新的数据（updatedAt 更大），应用启动时可能优先使用 localStorage 而非 JSON 文件。如遇此情况，清除 localStorage 或使用新的浏览器配置即可。
- Demo 数据不修改 Agent 逻辑、路由规则、意图识别或任何业务边界。

### 5.6 CI/测试环境覆盖路径

`demo:seed` 和 `demo:verify` 均支持通过 `DEMO_STATE_PATH` 环境变量覆盖 state 文件路径，用于 CI 或沙箱环境：

```bash
DEMO_STATE_PATH="/tmp/demo-state.json" npm run demo:seed
DEMO_STATE_PATH="/tmp/demo-state.json" npm run demo:verify
```

不设置该变量时，脚本使用默认的 Electron userData 路径。

```
src/shared/demo/demo-household-seed.mjs
```

相关脚本：

```
scripts/demo-seed.mjs     # 写入 Demo 数据
scripts/demo-verify.mjs   # 验证数据完整性
```

所有数据固定，不使用随机数。相同版本的 Seed 多次执行，生成的业务数据一致。
