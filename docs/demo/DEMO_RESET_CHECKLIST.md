# Demo Reset 人工验收清单

## 前置条件

- 应用已启动并处于开发模式（`npm run dev`）或设置了 `VITE_DEMO_MODE=true`
- 当前存在旧业务数据（非 Demo 数据）

## 验收步骤

### 1. 在旧数据状态下点击恢复

1. 打开应用，确认首页显示的是旧业务数据（如电费、狗粮等）
2. 打开设置面板（右上角齿轮图标）
3. 滚动到设置底部，应看到「演示数据」分区
4. 点击「恢复比赛演示数据」按钮
5. 应弹出确认对话框：「恢复后，当前消耗品、补货记录和对话状态将替换为比赛演示数据。当前数据会自动备份。」
6. 点击「恢复演示数据」按钮

**预期**：按钮显示「正在恢复…」，随后页面自动重新加载

### 2. 页面无需手工清理 Local Storage

1. 恢复完成后，页面自动重新加载
2. 首页应直接显示 Demo 数据（猫砂、洗衣液、抽纸等）
3. 不需要手动清除 Local Storage 或重启应用

**预期**：页面重载后立即显示 Demo 数据，无残留旧数据

### 3. 恢复后分类数量正确

1. 查看首页左侧或顶部的分类列表
2. 应显示 7 个分类

**预期**：卫生间、厨房、洗衣清洁、宠物用品、日常护理、饮品零食、其他用品

### 4. 宠物用品包含猫砂、猫粮、宠物擦脚湿巾

1. 进入「宠物用品」分类
2. 确认包含以下商品：
   - 猫砂
   - 猫粮
   - 宠物擦脚湿巾

**预期**：三个商品都存在，无重复

### 5. 猫砂显示上次补货 7 月 3 日

1. 点击「猫砂」进入商品详情
2. 查看最近一次补货记录

**预期**：日期为 2026 年 7 月 3 日，数量 2 袋，金额 68 元

### 6. 猫砂显示还剩约 8 天

1. 在猫砂商品详情页查看预计剩余时间
2. 或在首页查看猫砂的状态标签

**预期**：显示「快用完」状态，剩余约 8 天（基于参考日期 2026-07-12）

### 7. 洗衣凝珠不存在

1. 在首页搜索或浏览所有分类
2. 确认不存在「洗衣凝珠」

**预期**：洗衣凝珠不存在于任何分类中

### 8. 问问管家进入干净会话

1. 打开「问问管家」对话面板
2. 确认对话历史为空（无残留消息）
3. 应看到开场简报或空白状态

**预期**：对话面板为干净状态，无旧消息残留

### 9. API Key 和模型仍可正常调用

1. 打开设置面板
2. 确认 API Key 字段仍显示恢复前的 Key
3. 确认问答模型和订单识别模型仍为恢复前的配置
4. 在对话面板发送一条测试消息（如「猫砂还能用多久」）

**预期**：API Key 保留，模型配置保留，Agent 正常响应

### 10. 连续点击恢复不会重复写入

1. 再次打开设置，点击「恢复比赛演示数据」
2. 确认并等待恢复完成
3. 恢复后立即再次点击恢复
4. 每次恢复后商品数都应为 15

**预期**：连续恢复不会产生重复商品或重复补货记录

### 11. 应用重启后仍是标准 Demo State

1. 完全退出应用（Cmd+Q 或托盘退出）
2. 重新启动应用
3. 确认首页仍显示 Demo 数据

**预期**：重启后数据一致，仍为 15 个商品的标准 Demo State

### 12. 无明显控制台错误

1. 打开开发者工具（Cmd+Option+I）
2. 查看 Console 标签
3. 确认无红色错误

**预期**：无 JavaScript 错误、无数据加载失败

## 终端命令验收

```bash
npm run demo:reset
```

**预期输出**：

```
Demo reset started
Backup created
  .../demo-backups/state-before-demo-reset-XXXXXXXX-XXXXXX.json
Main state written
  .../reminder-state.json
Demo state verified
15 items
31 restock records
API Key preserved

Demo reset completed
```

## 备份验证

1. 执行恢复后，检查 `userData/demo-backups/` 目录
2. 确认有备份文件存在
3. 连续恢复 3 次以上后，确认备份文件不超过 3 份

**预期**：备份文件名为 `state-before-demo-reset-YYYYMMDD-HHMMSS.json`，最多保留 3 份
