# 403家庭管家 手动 QA 测试清单

> 本清单用于在共享文件、UI 优化、打包发布前，按统一流程对桌面应用进行手动回归测试。
> 目标：把当前产品核心规则和本地数据风险固化为可执行步骤，确保「不白屏、不丢数据、不写坏文件、核心补货逻辑正确」。
>
> 适用范围：`household-replenishment-desktop/`（Electron + React + TypeScript + Vite，macOS / Windows）。
>
> 本文件仅描述测试步骤与注意事项，不要求测试者真的删除个人数据。

---

## 1. 测试原则

- **不直接删除现有数据**：任何情况下都不允许通过删文件、清空 localStorage、运行清库脚本等方式直接抹掉用户真实数据。
- **测试前先备份**：每次测试开始前必须先备份当前数据文件，备份命名见第 2 节。
- **新用户测试需要清空到干净状态**：使用专用测试目录或专用用户数据目录，不要污染真实数据。
- **旧数据测试需要恢复备份**：测试结束后必须把数据恢复到测试前备份的状态，避免测试副作用残留。
- **坏数据测试需要用专门构造的数据**：在专用测试目录中手工构造脏数据样本，不要在真实数据上做破坏性测试。
- **不把 mock/demo 数据当成真实使用结果**：demo 物品仅用于验证渲染，不能作为功能验收依据。
- **测试目标**：不白屏、不丢数据、不写坏文件、核心补货逻辑正确。任何一项不满足都视为不通过。

## 2. 测试前准备

1. 确认当前分支是 `main`：

   ```bash
   git branch --show-current
   ```

2. 拉取最新代码：

   ```bash
   git pull origin main
   ```

3. 类型检查通过：

   ```bash
   npm run typecheck
   ```

4. 构建通过：

   ```bash
   npm run build
   ```

5. 记录测试环境信息（写入第 13 节模板）：
   - 测试日期与时间；
   - 操作系统及版本（macOS / Windows）；
   - 应用版本（`package.json` 中的 `version`）；
   - commit sha（`git rev-parse HEAD`）。

6. 找到当前应用数据文件 / 本地存储位置：
   - macOS 默认：`~/Library/Application Support/403家庭管家/`（或 Electron `app.getPath('userData')` 对应目录）；
   - Windows 默认：`%APPDATA%/403家庭管家/`；
   - 实际路径以 `electron/main.js` 中 `app.getPath('userData')` 为准。

7. 备份当前数据，命名格式：

   ```
   backup-before-qa-YYYY-MM-DD-HHmm
   ```

   例如：`backup-before-qa-2026-06-19-1430`。备份应包含整个 userData 目录，至少包含 state 文件与历史记录。

8. 注意：本清单只写测试步骤和注意事项，**不要求用户真的删除数据**。需要“干净状态”时，请使用独立的测试 userData 目录（通过环境变量或临时配置切换），不要动真实数据。

## 3. 新用户空数据测试

> 目标：验证从空数据启动时的首启体验，确保不会白屏、不会自动注入 demo 物品。

使用独立的空 userData 目录启动应用，按顺序验证：

1. 首次打开应用不白屏，主界面正常渲染。
2. 不自动注入 demo / mock 物品，列表为空。
3. 空状态文案正常显示（例如“还没有物品，添加一个开始管理吧”之类的引导文案）。
4. 可以从空状态入口（按钮 / 链接）进入新建物品界面。
5. 新建一个真实物品，例如名称填“抽纸”，分类、周期等按默认或合理值填写。
6. 保存后列表出现该物品，字段显示正确。
7. 关闭应用再打开，物品仍然存在，字段一致。
8. 删除该物品不会报错；再次编辑该物品也不会报错（先新建再编辑）。

**通过标准**：1–8 全部符合预期，无白屏、无报错、无 demo 残留。

## 4. 新建物品测试

> 目标：验证新建物品表单所有字段能正确写入 state，非法输入不会污染数据。

依次覆盖以下字段，每项保存后重启应用确认持久化正确：

- **名称 `name`**：必填，空名称应阻止保存或给出提示。
- **分类 `category`**：可选，能正确保存与显示。
- **周期 `cycleDays`**：正整数，例如 30；0、负数、字符串应被拒绝或降级处理。
- **提前提醒天数 `bufferDays`**：非负整数，例如 3；不能大于 `cycleDays`。
- **默认补货数量 `defaultQty`**：正数，例如 1；空值应按 1 处理。
- **单位 `unit`**：字符串，例如“包”“瓶”。
- **价格 `price`**：非负数字，可空。
- **平台 `platform`**：字符串，可空，例如“京东”“淘宝”。
- **购买链接 `link`**：必须是 `http://` 或 `https://` 开头；非 http(s) 链接应被拒绝。
- **无购买链接时也能正常保存**：`link` 留空时保存成功，列表显示正常。
- **非法输入不要导致 NaN / undefined 写入 state**：例如在数字字段输入字母、留空、粘贴带空格的内容，保存后检查 state 文件中对应字段不是 `NaN` / `undefined` / `null`（除非业务允许 null）。

**通过标准**：所有字段保存与持久化正确，state 文件中无 `NaN` / `undefined`。

## 5. 补货主流程测试

> 目标：验证补货是“直接补货”语义，不引入订单 / 待收货状态。

对至少一个有购买链接的物品和一个无购买链接的物品分别测试：

- **有购买链接时**：点击“去补货 / 去购买”应打开系统默认浏览器跳转到该链接，不在应用内打开。
- **没有购买链接时**：也能执行“已补货 / 补货”操作，仅更新本地补货记录。
- **补货后 `lastRestockedAt` 更新**：state 文件中该物品的 `lastRestockedAt` 更新为当前时间附近。
- **补货后状态回到“充足”**：UI 上该物品状态显示为“充足”。
- **补货后 `snoozeUntil` 清空**：state 文件中该物品的 `snoozeUntil` 被清空（null / undefined / 不存在）。
- **补货后不写入 `orderedAt`**：state 文件中不出现 `orderedAt` 字段，或字段被显式移除。
- **不出现“已下单 / 待收货 / 订单状态”**：UI、state、history 中均不出现这些字样或字段。
- **补货记录进入 history**：`history` 数组新增一条补货记录，包含时间、数量等。
- **重启后补货结果仍保留**：关闭应用再打开，`lastRestockedAt`、history、状态均与重启前一致。

**通过标准**：以上全部符合，且 state 文件中无 `orderedAt`、无订单相关字段。

## 6. 状态文案测试

> 目标：验证三类状态的可见文案统一，不出现被废弃的旧词汇。

覆盖三类状态：

- **充足**（ok / sufficient）；
- **快用完**（warning）；
- **预计已用完 / 急需补货**（urgent）。

要求：

- warning 状态用户可见文案是 **“快用完”**；
- urgent 状态用户可见文案是 **“预计已用完 / 急需补货”**；
- **不出现**以下词汇：到期、逾期、过期、截止、已下单、待收货、订单；
- 系统通知文案（macOS Notification Center / Windows 通知中心）也符合上述规则；
- 列表卡片、详情页、通知、空状态等所有用户可见位置均一致。

**通过标准**：三类状态文案正确，全应用无废弃词汇。

## 7. qty 周期学习测试

> 目标：验证补货数量 `qty` 参与消耗周期估算，且不影响订单语义。

测试方法：

1. 创建一个物品，周期 `cycleDays = 30`，`defaultQty = 1`。
2. 模拟一次补货，补货间隔约 30 天（可通过修改 `lastRestockedAt` 构造，或在专用测试目录中等待）。
3. 本次补货数量 `qty` 填 2。
4. 预期新的预计周期应接近 **60 天**（按 qty=2、间隔 30 天估算消耗速率）。
5. **qty 缺失时应按 1 处理**：补货记录不写 qty 或写 null 时，周期学习按 qty=1 计算。
6. **qty 为 0、负数、非法值时不能写坏数据**：state 中不出现 NaN / undefined，周期学习跳过该条或按 1 处理。
7. **`price` / `platform` 不参与周期学习**：修改价格或平台不影响预计周期。
8. **补货数量只影响消耗周期估算，不代表订单数量**：UI 与 state 中不出现“订单数量”“下单数量”等字样。

**通过标准**：qty=2 时预计周期接近 60 天；非法 qty 不写坏数据；price/platform 不影响周期。

## 8. 旧数据兼容测试

> 目标：验证历史数据（含已废弃字段）能被正确迁移或降级处理，不白屏。

恢复测试前备份或使用旧数据样本，验证：

- 旧 demo 数据不会导致白屏，能正常加载列表。
- 旧 `reminderDaysAhead` 可以迁移到 `bufferDays`（迁移后 `bufferDays` 有值，旧字段不再被读取或被显式移除）。
- 旧 `orderedAt` 不再进入新 state（迁移后 state 中无 `orderedAt`，或字段被忽略）。
- 缺失 `purchaseOptions` 的物品仍能补货（按无购买链接处理）。
- 旧 history 顺序正常，时间倒序 / 正序与现版本一致。
- 缺失 `qty` 的旧 history 按 1 处理，周期学习不报错。
- 保存一次、重启一次后数据仍正常，迁移结果稳定。

**通过标准**：旧数据加载不白屏，废弃字段被迁移或忽略，重启后状态稳定。

## 9. 坏数据 / 脏数据测试

> 目标：验证应用对损坏数据的容错能力，不白屏、不覆盖原文件、能备份 corrupt raw。

在专用测试目录中手工构造以下坏数据样本，逐项验证：

- **state 文件为空**（0 字节）；
- **state 文件是非法 JSON**（例如 `{name: "broken"` 缺引号、尾逗号、纯文本）；
- **item 缺少 `name`**；
- **item 缺少 `cycleDays`**；
- **`cycleDays` 是字符串 / 0 / 负数**；
- **`bufferDays` 大于 `cycleDays`**；
- **`snoozeUntilHour` 超出 0–23**（例如 -1、24、99）；
- **`purchaseOptions` 不是数组**（例如是对象、字符串、null）；
- **`history` 不是数组**；
- **`qty` 是空、0、负数、字符串**。

预期：

- **不白屏**：应用能启动并显示列表或兜底界面。
- **不直接覆盖损坏原文件**：损坏的原始文件保留在原位或被备份。
- **能备份 corrupt raw**：应用将损坏内容备份为 `corrupt-raw-YYYY-MM-DD-HHmm.json` 之类文件。
- **可恢复的数据尽量保留**：例如只有某个 item 字段损坏时，其他 item 不受影响。
- **不写入 NaN / undefined**：state 文件中不出现 `NaN` / `undefined` 字面量。
- **UI 能提示异常或降级**：例如显示“数据异常，已备份原始文件”之类的提示。

**通过标准**：所有坏数据样本均不白屏、不覆盖原文件、有 corrupt raw 备份、无 NaN/undefined。

## 10. 持久化测试

> 目标：验证读写流程不会丢数据、不会写坏文件。

覆盖：

- **新建后重启仍存在**：新建物品 → 关闭应用 → 重开 → 物品仍在。
- **编辑后重启仍保留**：编辑物品字段 → 重启 → 字段为编辑后的值。
- **补货后重启仍保留**：补货 → 重启 → `lastRestockedAt`、history、状态均保留。
- **清空列表后重启仍为空，不重新注入 demo**：手动清空所有物品 → 重启 → 列表仍为空，不出现 demo 物品。
- **写入失败时不应覆盖旧文件**：模拟写入失败（例如磁盘只读、权限不足、目录不存在）时，旧 state 文件不被覆盖。
- **解析失败时不应直接清空用户数据**：state 文件解析失败时，应用不写入空 state 覆盖原文件，而是保留原文件并备份。

**通过标准**：以上全部符合，无数据丢失、无文件被错误覆盖。

## 11. Electron 安全与外链测试

> 目标：验证外链打开行为安全可控，不在应用内新开窗口。

覆盖：

- **购买链接只允许 http/https**：`link` 字段以 `http://` 或 `https://` 开头才能被打开。
- **非 http/https 链接不打开**：例如 `javascript:`、`file:`、`data:`、`vbscript:` 等协议被拒绝，不执行。
- **外链通过系统浏览器打开**：点击购买链接时调用 `shell.openExternal`，在系统默认浏览器中打开。
- **不在应用内新开窗口**：不创建新的 `BrowserWindow` 加载外链。
- **`window.open` / `target=_blank` 不绕过限制**：渲染进程中的 `window.open` 或 `<a target="_blank">` 不直接加载外链，统一走 `shell.openExternal`。
- **应用内路由不被外部链接劫持**：尝试通过外链跳转到应用内路由（例如 `file:///.../index.html#/dashboard`）不会篡改应用状态。

**通过标准**：所有外链均通过系统浏览器打开，非 http(s) 链接被拒绝，应用内不新开窗口。

---

## 12. Error Boundary / 崩溃兜底测试

> 目标：验证渲染异常时有兜底界面，不无限白屏、不清空数据。

覆盖：

- **构造一个渲染异常时，应用显示兜底界面**：例如临时在某个组件抛出异常（仅在测试构建中），验证 ErrorBoundary 捕获并显示兜底 UI。
- **不无限白屏**：兜底界面正常渲染，不是空白页。
- **主进程异常有日志**：主进程 `uncaughtException` / `unhandledRejection` 写入日志，不在控制台静默丢失。
- **`unhandledRejection` 不直接导致无提示崩溃**：渲染进程的 Promise 异常被捕获或提示，不直接闪退。
- **兜底界面不清空用户数据**：触发兜底界面后，state 文件内容与触发前一致，不写入空数据。

**通过标准**：渲染异常有兜底界面，主进程异常有日志，数据不被清空。
