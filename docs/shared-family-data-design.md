# 共享家庭数据文件方案设计

> 文档用途：冻结「夫妻跨设备共享家庭消耗品补货数据」的实现边界，先设计后实现，不做账号系统和云服务。
>
> 当前状态：设计中，未实现

---

## 1. 目标

本方案要解决一个具体问题：

* 夫妻两人共同维护同一份家庭消耗品清单（卫生纸、洗衣液、猫粮等）；
* 两个人使用不同的电脑（一台 Windows、一台 macOS）；
* 任何一方记录补货后，另一方能看到最新的状态；
* 不引入账号系统、服务器、数据库、云端同步服务；
* 尽量保留当前"本地优先、隐私友好"的产品形态——数据仍然是一个普通文件，用户自己决定如何备份和同步。

简单来说：

> 应用把 state 写到一个 JSON 文件，两个人把这个文件放在同一个云盘/局域网同步目录里。两台电脑各自读写同一个文件。由用户已经在用的同步工具（iCloud Drive / OneDrive / Dropbox / 坚果云 / Syncthing / NAS / SMB）负责在两台电脑之间同步文件。

---

## 2. 非目标

本阶段明确不做以下事情。这些事项后续可以单独再评估，但不应在第一版实现中出现：

* **不做账号系统**：没有"登录"、没有"用户中心"、没有用户名/密码；
* **不做后端服务**：没有家庭管理后台、没有 API、没有云端数据库；
* **不做实时协同编辑**：不支持多人同时看到对方的编辑；不做 Operational Transform / CRDT；
* **不做多人权限管理**：不区分"主账户"、"成员"、"只读"等角色；每个家庭成员都能读写；
* **不做冲突自动智能合并**：不尝试在字段级别自动合并两个版本；冲突时让用户选择；
* **不做移动端**：当前优先支持桌面 macOS / Windows，不在本方案里引入 iOS / Android；
* **不做云厂商绑定**：不限制到某一个云盘；不针对某一个云盘做特殊适配；
* **不做端到端加密实现**：可以在文档中作为未来改进项列出，但不在本方案实现；
* **不改现有数据结构**：共享文件的 JSON schema 应尽量从当前 state 自然延伸，不做结构性重构。

---

## 3. 推荐方案

核心思路：**一个共享 JSON 文件 + 用户自己选择的同步目录**。

### 3.1 架构图（文字版）

```
┌──────────────────┐           ┌────────────────────────────┐           ┌──────────────────┐
│ 电脑 A (macOS)   │           │ 用户选择的同步目录          │           │ 电脑 B (Windows) │
│                  │           │                            │           │                  │
│ 403 家庭管家     │──读取/──▶│ household-replenishment.    │◀──读取/──│ 403 家庭管家     │
│                  │   写入    │ family.json                │   写入    │                  │
└──────────────────┘           └────────────────────────────┘           └──────────────────┘
                                        ▲
                                        │
                           由用户自己的同步工具负责：
                      iCloud Drive / OneDrive / Dropbox /
                      坚果云 / Syncthing / NAS / SMB / ...
```

### 3.2 方案要点

* **单文件共享**：所有家庭共享数据保存在一个 JSON 文件中，文件名建议为 `household-replenishment.family.json`；
* **用户选择存储位置**：应用在设置页提供一个"选择共享文件位置"的操作。用户可以选择：
  * iCloud Drive 中的某个目录；
  * OneDrive 中的某个目录；
  * Dropbox 中的某个目录；
  * 坚果云同步目录；
  * NAS / SMB 共享目录；
  * Syncthing 同步目录；
  * 任何一个普通本地文件夹（用于测试或本地备份）。
* **两台电脑指向同一个文件**：家庭成员 A 和家庭成员 B 都在各自的应用设置里选择同一个同步目录下的同一个 JSON 文件；
* **应用启动时加载这个文件**：每次启动时，从这个共享文件中读取家庭 state；
* **每次保存时写回这个文件**：用户做了补货、编辑等操作后，应用把 state 写回同一个共享文件；
* **默认仍然是本地文件**：如果用户没有设置共享文件路径，应用继续使用当前的本地 state 路径，行为和现在完全一致。

## 4. 文件格式建议

### 4.1 顶层结构

共享文件的建议格式如下。这只是设计建议，具体字段需要在后续实现中从当前 `state` 平滑迁移：

```jsonc
{
  "schemaVersion": 1,
  "updatedAt": 1718000000000,
  "deviceId": "device-mac-pro-2024-06-19",
  "items": [
    {
      "id": "item-xxx",
      "name": "卫生纸",
      "category": "卫生间",
      "cycleDays": 30,
      "bufferDays": 5,
      "lastRestockedAt": 1718000000000,
      "anchorEstimated": false,
      "history": [
        {
          "id": "rec-xxx",
          "at": 1718000000000,
          "intervalDays": 28,
          "price": 59.9,
          "qty": 2,
          "platform": "淘宝"
        }
      ],
      "purchaseOptions": [],
      "price": 59.9,
      "unit": "提",
      "platform": "淘宝",
      "defaultQty": 2,
      "suggestedCycleDays": null,
      "learningEnabled": true
      // ... 其他现有字段保持不变
    }
  ],
  "settings": {
    "reminderIntervalMinutes": 60,
    "quietStart": "22:00",
    "quietEnd": "08:00",
    "snoozeUntilHour": 8
  },
  "meta": {
    "lastModifiedByDeviceId": "device-mac-pro-2024-06-19",
    "deviceName": "Mac Pro",
    "createdAt": 1718000000000,
    "comment": "403家庭管家 - 家庭共享补货数据"
  }
}
```

### 4.2 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `schemaVersion` | `number` | 版本号，用于后续迁移。从 1 开始。 |
| `updatedAt` | `number` | 最后更新时间戳（毫秒）。用于冲突比较和时序判断。 |
| `deviceId` | `string` | 最后写入该文件的设备标识（详见第 8 节）。 |
| `items` | `array` | 家庭消耗品清单，字段保持和当前 `state.items` 一致。 |
| `settings` | `object` | 可共享的应用设置，如提醒间隔、静音时段等；不包含每台设备自己的本地文件路径。 |
| `meta` | `object`（可选） | 额外元信息，如设备名称、创建时间等，用于调试和提示。 |

### 4.3 迁移注意

* 当前 `state` 的结构可以基本原样套入上述共享文件的 `items` 和 `settings` 部分；
* 不要在设计阶段强制用户重构现有 schema；
* 后续实现时应该支持"从当前本地 state 平滑迁移到共享文件"，不需要用户手动迁移；
* **共享文件路径保存在每台设备自己的本地配置里，不写入家庭共享 JSON 文件**——两台设备的 `sharedFilePath` 可能完全不同（Windows 的 iCloud 路径 vs macOS 的 iCloud 路径），如果写入共享文件，两台设备会互相覆盖对方的路径。

## 5. 写入策略

### 5.1 安全写入流程

写入共享文件必须保证：在网络延迟或云盘正在同步的情况下，不会把文件写坏。建议采用以下三步：

1. **先写临时文件**：把新的 state 写到 `<目标文件名>.tmp.<deviceId>.<时间戳>.json`；
2. **尽量保证落盘**：如果 Node.js / Electron 环境允许，调用 `fsync` 或同等效果的操作，让操作系统把缓存真正写到磁盘；
3. **原子替换**：把临时文件 `rename` 到正式文件名。

这和 PR #6 主进程 state 写入兜底思路是一致的：在共享文件场景下，这个策略尤为重要，因为两台设备的云盘同步可能在任意时刻发生。

### 5.2 失败处理

* **写入失败保留旧文件**：只要正式文件名的内容还没被替换，用户就还有上次的有效数据；
* **写入失败要提示用户**：弹出一条可见的错误，告诉用户"刚才的保存失败了，请检查磁盘/网络/路径"；
* **不要直接覆盖损坏文件**：如果磁盘上的文件已经损坏（比如 JSON 解析失败），不要自动覆盖它——先让用户看到"文件损坏"并给出备份选项。

### 5.3 写入频率

* **每次用户做了真正的数据变更时才保存**：不要在无变更时反复写磁盘；
* **防抖 / 合并**：用户连续操作时（例如快速连续点击补货），合并为一次写入，比如 300-500ms 内只写一次，减少磁盘 IO 和云盘同步冲突。

## 6. 读取策略

### 6.1 启动时读取

应用启动时按以下顺序决定读取哪个数据文件：

1. 如果用户已在设置里指定了共享文件路径 → 读取该共享文件；
2. 如果没有设置共享文件路径 → 继续使用当前本地 state（和现在完全一致）；
3. 如果共享文件路径设置了，但文件不存在 → 提示用户："在该位置找不到共享文件。你是第一次使用共享，还是之前的共享文件被移动了？"

### 6.2 文件不存在的处理

* 如果用户是**第一次启用共享**，应用应该问："是否从当前本地数据创建一个新的家庭共享文件？"；
* 如果用户**曾经有共享文件但现在路径无效**，应用应该给两个选项：
  * "选择另一个文件路径"；
  * "断开共享，回到本地数据"。

### 6.3 文件损坏的处理

读取到无法解析的 JSON 或校验失败的内容时：

1. **不要覆盖它**，不要用空的/默认的 state 覆盖磁盘上损坏的文件；
2. **把损坏的原始内容另存为备份**：`<原文件名>.corrupt.<时间戳>.json`，保留给用户手动恢复；
3. **提示用户**：告诉用户"家庭共享数据文件损坏，已备份。你可以从另一台设备重新同步，或选择其他文件路径"；
4. **给选项**：使用最近一次有效备份（如果应用保留本地备份副本），或者让用户手动选择其他文件。

### 6.4 外部文件变化的监听

第一版可以**不做实时监听**。简单做法：

* 应用启动时读取一次；
* 用户每次主动操作后写回；
* 如果另一台设备更新了文件，当前设备需要关闭后再打开，或者在 UI 上有一个"刷新"按钮。

后续版本可以考虑：

* `fs.watch` 监听共享文件变化；
* 轮询文件 mtime / size，变化了则提示用户"有新数据，是否刷新"；
* 但第一版不要做，先验证整体方案是否能跑通。

## 7. 冲突策略

### 7.1 不做实时协同

家庭消耗品补货是低频场景（每天通常 0-2 次操作），两人同时编辑的概率很低。第一版不做：

* 不做 Operational Transform；
* 不做 CRDT；
* 不做字段级自动合并；
* 不做实时同步提示。

### 7.2 单写者优先的简单规则

* 家庭里尽量不要两个人同时在应用里编辑；
* 两人错开使用时，云盘/局域网同步工具会自动把文件同步过去，不需要额外处理；
* 只在"磁盘文件和内存状态不一致"时做一次冲突检查。

### 7.3 写入前检查

在真正写回共享文件前，做一次简单的时序比较：

1. 读取磁盘上共享文件的 `updatedAt`；
2. 比较应用内存里的 `updatedAt`（或上一次从文件读到的时间戳）；
3. 如果磁盘文件比应用内存新，说明"另一台设备已经改过这个文件"，进入冲突提示；
4. 如果磁盘文件和内存的 `updatedAt` 一致或更旧，说明没有外部变化，可以正常写入。

### 7.4 冲突提示

当检测到磁盘文件和应用内存不一致时，弹出一个简单对话框给用户三个选项：

1. **使用磁盘版本**——丢弃本机未保存的变更，加载另一台设备刚写入的数据；
2. **使用本机版本覆盖**——把本机当前 state 写回文件，另一台设备下次启动时会看到这个新版本；
3. **另存为备份文件**——把本机当前 state 保存到 `<原文件名>.conflict.<deviceId>.<时间戳>.json`，让用户后续手动对比。

### 7.5 云盘自己的冲突文件

OneDrive / iCloud Drive 等同步工具本身可能在两台设备同时写入时生成冲突副本（例如 `household-replenishment.family (Mac Pro 的冲突副本).json`）。应用不需要对这些文件做自动处理，但可以：

* 在启动时检测到类似文件名，提示用户"检测到云盘冲突副本，建议手动检查"；
* 不让这些文件参与正常读写流程。

## 8. 设备标识

### 8.1 需要设备标识的理由

* 两台设备都能写同一个文件，写入时记录"是谁写的"；
* 冲突提示时能告诉用户"另一台设备刚刚改过"，而不是匿名的"有冲突"；
* 临时文件命名中带设备 ID，能避免两台设备的临时文件互相干扰。

### 8.2 生成方式

* 每台设备第一次启用共享功能时，在本地生成一个 `deviceId`；
* 建议格式：`device-<主机名或简化设备名>-<第一次启用日期>`，例如 `device-Mac-Pro-2024-06-19`；
* 或者用随机 ID（如 `crypto.randomUUID()`），再配上用户可编辑的昵称；
* `deviceId` 只保存在本机，不上传服务器，不用于账号系统。

### 8.3 使用方式

* 写入共享文件时，把 `deviceId` 记录在文件顶层的 `deviceId` 字段和 `meta.lastModifiedByDeviceId` 字段；
* 冲突提示时，如果文件里的 `deviceId` 和本机 `deviceId` 不同，提示："您的另一台设备刚刚更新了家庭共享数据。您是否要切换到最新版本，还是保留本机的编辑？"

## 9. 用户流程

### 9.1 设置页新增"家庭共享数据文件"区块

在应用设置页增加一个新的区块。内容包含：

* **当前状态**：显示"未启用共享"或"已启用共享（文件路径：xxx）"；
* **上次同步时间**：如果已启用共享，显示"上次写入时间"和"上次从文件读取时间"；
* **操作按钮**：
  * "选择共享文件位置..."——打开文件选择对话框；
  * "断开共享，改用本机数据"——把共享路径清空，回到本地模式；
  * "导出当前数据为家庭共享文件..."——把当前本地数据导出到用户指定的新位置，方便第一次同步。

### 9.2 第一次启用共享的流程

```
用户打开设置 → 找到"家庭共享数据文件"
         ↓
点击"选择共享文件位置..."
         ↓
系统文件对话框弹出，用户选择：
  • iCloud Drive 里的目录 + 输入新文件名
  • 或 OneDrive / Dropbox / 坚果云 / Syncthing / NAS / SMB 等目录
         ↓
应用判断：
  • 如果该路径已有合法的共享文件 → 加载该文件的数据
  • 如果该路径为空 / 不存在 → 提示"是否用当前本地数据创建家庭共享文件？"
         ↓
确认后 → 写入共享文件 → 显示"已启用家庭共享，文件位于 xxx"
         ↓
在另一台设备上重复同样流程，选择同一个文件路径
```

### 9.3 日常使用

* 用户正常使用应用——记录补货、编辑物品、调整补货周期；
* 每次保存后，应用写回共享文件（按第 5 节安全写入策略）；
* 用户关闭应用，云盘/局域网同步工具把文件同步到另一台设备；
* 另一台设备的用户打开应用 → 读取最新文件（按第 6 节读取策略）。

### 9.4 断开共享

用户点击"断开共享，改用本机数据"时：

1. 应用把当前共享文件里的 state 复制一份到本地；
2. 在本地设置里清空共享文件路径；
3. 显示"已断开家庭共享，使用本机数据"的提示；
4. 磁盘上的共享文件不删除——留给另一台设备继续使用。

## 10. 风险和限制

在文档中必须清晰地告诉用户和后续开发者：本方案有以下已知限制。

| 风险 | 说明 | 对策 |
|------|------|------|
| **云盘同步有延迟** | iCloud Drive / OneDrive 等不会在文件变更后立即把内容同步到另一台设备 | 文档中提醒用户"等同步完成后再打开另一台设备"；第一版不强求实时监听 |
| **两台设备同时编辑可能冲突** | 如果两人在一两分钟内都做了编辑，第二次写入的一方可能覆盖第一次 | 写入前 `updatedAt` 检查 + 冲突提示（第 7 节）；尽量错开使用 |
| **云盘自己可能生成冲突副本** | OneDrive / iCloud 有时会自动生成 `xxx (Mac Pro 的冲突副本).json` | 检测并提示用户手动检查；不做自动合并 |
| **用户需要自己管理云盘权限** | 谁有权限看/改这个文件，是用户自己在云盘/局域网里配置的 | 应用不做权限管理；文档中明确这一点 |
| **文件路径失效时要降级** | 云盘登录过期、网络断开、NAS 未挂载等都可能导致共享路径无法访问 | 检测到路径不可写时，提示用户"共享文件路径不可访问，当前编辑暂存到本地"，路径恢复后再同步 |
| **Windows / macOS 路径差异** | 同一同步目录在两台设备上路径完全不同（如 iCloud 的路径 macOS 有特殊格式） | 每台设备各自设置路径，不依赖对端路径；路径只存在本地设置里 |
| **JSON 文件不适合高频实时协同** | 如果两个人一天同时打开应用并频繁操作，冲突概率会升高 | 本方案适合低频补货场景；不适合多人实时协作（见第 2 节"不做实时协同编辑"） |
| **数据备份由用户承担** | 应用不做自动云备份，数据安全性取决于用户的云盘/局域网设置 | 文档中明确提醒用户定期自己备份；应用可以提供"手动导出/导入"功能 |

总结：

> 本方案适合"一对夫妻、低频补货记录"的家庭场景。它不是一个通用的多人实时协作方案。

## 12. 决策建议

给本方案一个明确的决策结论，供后续开发参考：

### 推荐：采用"共享 JSON 文件 + 云盘/局域网同步目录"方案

理由：

1. **零新增依赖**：不需要引入后端、数据库、账号系统；
2. **零持续成本**：不需要服务器、不需要云服务订阅、不需要维护者长期运维；
3. **隐私友好**：数据始终是用户自己的文件，存在用户自己的存储里；
4. **兼容当前架构**：当前已经是 JSON 文件状态，只是换一个路径 + 增加一个文件选择 UI + 加固读写策略；
5. **用户已经在用云盘**：用户已经有 iCloud Drive / OneDrive 等同步工具，不需要学习新工具或注册新服务；
6. **实现可控**：可以按第 11 节拆成多个小 PR，每个 PR 独立可验证、独立可回滚。

### 不建议现在做的事

* **不建议现在做账号系统**——对于两人家庭场景，账号系统是过度设计，引入登录、找回密码、隐私政策、合规等一系列与当前目标无关的工作；
* **不建议现在做自建云同步**——自己维护一个同步服务是长期运维负担，而且对家庭场景而言没有必要；
* **不建议现在做实时协同（CRDT / OT）**——家庭补货是低频场景，冲突概率很低，不值得在这个阶段引入复杂算法和状态管理。

### 下一步动作

1. **本 PR 合并后，冻结设计边界**；
2. **按第 11 节拆分的 PR 顺序，实现最小可用版本**（PR A → PR B → PR C）；
3. **在两个人的设备上手动测试完整流程**：一人创建共享文件，另一人加入同一文件，验证补货操作能被对方看到；
4. **测试通过后，再考虑要不要继续做 PR D 及之后的冲突处理**。
